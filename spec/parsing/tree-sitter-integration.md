# Limelight‑X Tree‑sitter Integration Architecture  
## Using `tree-sitter-limelightx.dll` Directly in Avalonia (.NET)

This document describes the architecture for integrating the Limelight‑X Tree‑sitter grammar DLL  
(`tree-sitter-limelightx.dll`) directly into the Avalonia editor.  
It is intended for Coding Assistants and future maintainers of the Limelight‑X CNL editor.

This document is subordinate to `spec/cnl-editor-architecture.md`, the parent authority for the Tree‑sitter integration (two-independent-parsers model, client‑side‑only scope, three-tier file model). Where this document and that one would otherwise conflict, `spec/cnl-editor-architecture.md` wins. `ui/native/<rid>/tree-sitter-limelightx.dll` (`<rid>` = `win-x64` or `win-arm64`) and `ui/queries/*.scm` are tier 3 of that document's §3.1 three-tier file model; only the `win-arm64` DLL is currently built (`spec/parsing/tree-sitter-build-guide.md` §0, `CLAUDE.md` §3.5).

---

# 1. Overview

Tree‑sitter provides incremental parsing, syntax trees, error nodes, and query execution.  
The Limelight‑X grammar is compiled into a **self‑contained native DLL**, one per architecture:

```
ui/native/win-arm64/tree-sitter-limelightx.dll
ui/native/win-x64/tree-sitter-limelightx.dll   (pending - see tree-sitter-build-guide.md §9)
```

This DLL contains:

- The full Tree‑sitter runtime (statically linked)  
- The Limelight‑X grammar rules  
- The exported language symbol:  
  ```
  const TSLanguage * tree_sitter_limelightx();
  ```

Please read tree-sitter-build-guide.md for instructions on how to build this dll.  Once built copy it to the ui/native folder

Avalonia loads this DLL via P/Invoke and uses it to:

- Parse CNL text incrementally  
- Provide IntelliSense context  
- Provide syntax highlighting  
- Provide folding  
- Provide structural navigation  
- Provide error surfaces  

The Rust backend remains the authoritative compiler and receives **raw text**, not CST.

# 2. Architecture Diagram

Two independent, never-bridged paths — not a pipeline. This mirrors `spec/cnl-editor-architecture.md` §2 exactly; see that document for the canonical version.

```
+------------------------+          +------------------------+
|      Avalonia UI       |          |      Avalonia UI       |
|  - TextEditor           |          |  - AST/IR/Result Panels |
|  - IntelliSense          |          |  - Error Banner          |
|  - Highlighting          |          +-----------+------------+
|  - Folding               |                      |
+-----------+------------+                        | POST /run,/explain,/trace
            |                                      | ws://.../events (spec/api.md)
            v                                      v
+------------------------+          +------------------------+
| tree-sitter-limelightx.dll |      |  /src/api (Rust, HTTP+WS) |
|  - TSLanguage* loader        |    |  unchanged, per spec/api.md |
|  - Parser runtime            |    +-----------+------------+
|  - Query engine              |                |
+------------------------+                       v
                                     +------------------------+
                                     |   Rust Backend (CNL)   |
                                     |  - PEG parser          |

                                     +------------------------+
```

The DLL is never in the path to Rust — it is consumed only by the left column, entirely in-process within Avalonia.

# 4. Loading the DLL in Avalonia

Avalonia loads the DLL using P/Invoke.

## 4.1 Language Loader

```csharp

[DllImport("tree-sitter-limelightx.dll")]
public static extern IntPtr tree_sitter_limelightx();
```

## 4.2 Parser Creation

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern IntPtr ts_parser_new();


[DllImport("tree-sitter-limelightx.dll")]
public static extern bool ts_parser_set_language(IntPtr parser, IntPtr language);

[DllImport("tree-sitter-limelightx.dll")]
public static extern IntPtr ts_parser_parse_string(
    IntPtr parser,
    IntPtr oldTree,
    string input,
    UIntPtr length);
```

## 4.3 Parser Wrapper

```csharp
public class LimelightXParser
{
    private readonly IntPtr _parser;
    private readonly IntPtr _language;

    public LimelightXParser()
    {
        _parser = ts_parser_new();
        _language = tree_sitter_limelightx();
        ts_parser_set_language(_parser, _language);
    }

    public IntPtr Parse(string text)
    {
        return ts_parser_parse_string(
            _parser,
            IntPtr.Zero,
            text,
            (UIntPtr)text.Length);
    }
}
```

# 6. Query Engine (Highlights, Folds, Injections)

Your `.scm` files are embedded as strings.

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern IntPtr ts_query_new(
    IntPtr language,
    string source,
    UIntPtr length,
    out UIntPtr errorOffset,
    out TSQueryError error);
```

Cursor:

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern IntPtr ts_query_cursor_new();

[DllImport("tree-sitter-limelightx.dll")]
public static extern void ts_query_cursor_exec(
    IntPtr cursor,
    IntPtr query,
    TSNode node);
```

This powers:

- syntax highlighting  
- folding  
- prompt hole detection  
- IntelliSense context

# 7. IntelliSense Architecture

Tree‑sitter provides:

- cursor‑aware grammar context  
- valid‑next‑token prediction  
- error nodes  
- structural selection  
- variable discovery (a local scan of prior `bind_stmt` nodes in the CST)  
- pronoun reference preview (a local scan for the nearest preceding sentence)  
- folding regions  

Variable discovery and pronoun reference preview are **syntactic, CST-only, best-effort local echoes** — not semantic resolution. They happen to be reliable because CNL v0.1 has no branching (`spec/cnl-grammar.md` §7 Non-Goals: no conditionals/loops), so "nearest preceding sentence" always agrees with what Rust's AST Normalizer would resolve. They are never authoritative and are always superseded by `/explain`'s actual response — see `spec/cnl-editor-architecture.md` §1.1.3.

The Rust backend provides:

- AST normalization  
- pronoun resolution  
- IR generation  
- execution  

Tree‑sitter is **editor‑only**.  
Rust is **compiler‑only**.

# 8. Packaging the DLL

Place the DLL in the folder matching its architecture:

```
ui/native/win-arm64/tree-sitter-limelightx.dll
ui/native/win-x64/tree-sitter-limelightx.dll
```

`LimelightX.UI.csproj` resolves the matching folder automatically (the explicit publish RID when pinned, otherwise the host machine's own OS architecture) and copies it, flattened, into the output root - see `spec/parsing/tree-sitter-build-guide.md` §0 and `LimelightX.UI.csproj`'s `<None Include="native\$(_TreeSitterRid)\...">` items.

Avalonia loads it at runtime via P/Invoke.

- Avalonia loads the DLL via P/Invoke.

# 10. Memory Management

Tree‑sitter allocates memory for:
- Parsers  
- Trees  
- Queries  
- Query cursors  

These must be freed explicitly to avoid leaks inside the Avalonia process.

## 10.1 Freeing Trees

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern void ts_tree_delete(IntPtr tree);
```

Trees should be freed:
- after each parse cycle  
- when replacing an old tree with a new one  
- when closing a document  

## 10.2 Freeing Parsers

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern void ts_parser_delete(IntPtr parser);
```

Parsers are long‑lived objects but must be freed when the editor shuts down.

## 10.3 Freeing Queries

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern void ts_query_delete(IntPtr query);
```

Queries should be freed after each highlight/fold/injection pass.

## 10.4 Freeing Query Cursors

```csharp
[DllImport("tree-sitter-limelightx.dll")]
public static extern void ts_query_cursor_delete(IntPtr cursor);
```

Cursors are cheap but must be freed after each query execution.

# 11. Error Handling

Tree‑sitter provides structured error information during query creation.

## 11.1 Query Errors

```csharp
TSQueryError error;
UIntPtr errorOffset;

var query = ts_query_new(
    _language,
    source,
    (UIntPtr)source.Length,
    out errorOffset,
    out error);
```

If `error != TSQueryError.None`, the `.scm` file contains a syntax error.

The Coding Assistant must:
- surface the error in the editor console  
- include the offset  
- avoid executing the query  

## 11.2 Parse Errors

Tree‑sitter does not throw exceptions.  
Instead, it inserts `ERROR` nodes into the CST.

The IntelliSense engine must:
- detect `ERROR` nodes  
- surface diagnostics  
- avoid blocking Rust backend execution

# 12. Embedding `.scm` Files

Highlight, fold, and injection queries are stored in `.scm` files.

Recommended embedding pattern:

```csharp
internal static class HighlightQueries
{
    public static readonly string Source =
        File.ReadAllText(Path.Combine(AppContext.BaseDirectory, "queries", "highlights.scm"));
}
```

Rules:
- `.scm` files must be UTF‑8  
- `.scm` files must be copied to output directory  
- Coding Assistant must not inline `.scm` content unless explicitly asked  

---

# 13. Query Lifecycle

Each IntelliSense cycle should:

1. Parse text → `TSTree`  
2. Extract root node  
3. Create query  
4. Create cursor  
5. Execute cursor  
6. Process matches  
7. Free cursor  
8. Free query  
9. Free tree  

This ensures no native memory leaks.

---

# 14. ABI Stability

The DLL exports:
```c
const TSLanguage * tree_sitter_limelightx();
```

This symbol must remain stable across versions.

Rules for grammar evolution:
- Do not rename the exported symbol  
- Do not change calling conventions  
- Do not remove required rules from grammar.js  
- Do not introduce scanner.c unless necessary  
- Keep grammar changes backward‑compatible  

If ABI changes are required:
- bump DLL version  
- update P/Invoke bindings  
- update Coding Assistant onboarding  

---

# 15. Versioning Strategy

Recommended versioning:

```
tree-sitter-limelightx-v1.dll
tree-sitter-limelightx-v2.dll
```

Rules:
- Increment version when grammar.js changes  
- Increment version when `.scm` files change  
- Increment version when ABI changes  
- Store version in `/ui/native/`

# 16. Coding Assistant Integration Rules

Coding Assistants must:

- Use P/Invoke bindings exactly as documented  
- Never assume TreeSitterSharp is available  
- Never reference external runtimes  
- Never modify grammar.js without user instruction  
- Never generate scanner.c unless grammar requires it  
- Always free trees, queries, and cursors  
- Always treat Rust backend as authoritative  

Coding Assistants may:
- generate IntelliSense logic  
- generate highlight/fold engines  
- generate query runners  
- generate editor services
